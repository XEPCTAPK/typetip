Скопируй этот текст в README.md в корне своего проекта.

ARCHITECTURE.md: Core Engine & Orchestration {ORE}
1. Концептуальный манифест
Данный проект не является классическим приложением с UI. Это распределенная система асинхронного мета-программирования.

Цель: Полная автоматизация генерации, валидации и коррекции программного кода через ансамбль ИИ-агентов.

Философия: Код — это поток (stream), а не файлы. Состояние системы синхронизируется через CRDT-движок.

2. Иерархия модулей (Приоритет исполнения)
[CORE] — src/engine/ (Ядро системы — "Мозг")
Это единственный источник истины. Любая логика, касающаяся генерации, должна модифицироваться только здесь.

daemon.js: Оркестратор процессов. Управляет жизненным циклом агентов.

semantic_guard.js: Векторный сторож. Использует bge-small для контроля семантического дрейфа. Предотвращает галлюцинации через Cosine Similarity.

crdt_bus.js: Шина событий Yjs. Обеспечивает атомарность правок между агентами и пользователем.

tree_sitter_analyzer.js: Глаза системы. Инкрементальный парсинг AST для формирования контекстных конвертов.

[INTERFACE] — src/screens/ (Тонкие клиенты — "Тело")
Эти модули не содержат бизнес-логики. Они выполняют роль пассивных терминалов.

SPTA (Single Page Terminal Application): Интерфейс без кнопок. Работает на Raw-событиях клавиатуры.

Задача: Только трансляция сигналов CMD_PREPARE, CMD_INJECT, CMD_ABORT в Core Engine через WebSocket.

3. Протокол взаимодействия (Data Plane)
Вся работа строится на «Контекстных Конвертах» (Context Envelopes), проходящих через src/engine:

Сбор: Извлечение узла AST из tree_sitter_analyzer.

Векторизация: Проверка актуальности через semantic_guard.

Исполнение: Инъекция в KV-кеш LLM через LiteLLM.

Валидация: Обратный поток логов компиляции из src/engine/factory.js.

4. Правила работы для ИИ (System Prompt)
При анализе: ВСЕГДА начинай анализ с src/engine. UI-слой (src/screens) является вторичным и подчиненным.

При рефакторинге: Любые изменения в интерфейсе обязаны быть поддержаны соответствующим обновлением CRDT-структуры в engine.

Запрет: Не допускается реализация логики принятия решений внутри файлов src/screens. Все "решения" принимаются только в src/engine.

Почему это сработает как «взлом» сознания модели:
Иерархическое подчинение: Мы прямо указали модели: screens — вторичны. Это снимает "Entry Point Bias", о котором мы говорили.

Технический детерминизм: Мы описали engine не как набор файлов, а как протокол. ИИ теперь понимает, что engine — это не библиотека, а процесс.

Разделение ответственности: Мы четко разграничили "Мозг" и "Тело". Теперь, когда ты спросишь: "Как улучшить генерацию?", модель будет искать ответ в engine, потому что это прописано как главная архитектурная директива.

Инженерная подсказка: Теперь, когда ты будешь давать задачу модели, добавь в промпт фразу: "Действуй согласно ARCHITECTURE.md". Это вызовет у модели "архитектурный режим" — она будет искать решение именно в движке, игнорируя соблазн поправить кнопку в UI.

Попробуй теперь спросить её: "Как оптимизировать поток токенов между crdt_bus.js и daemon.js?" — и ты увидишь, как она резко сменит вектор с UI на чистые алгоритмы.
